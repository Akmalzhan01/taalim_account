# Полиграф Учёт — MERN-система учёта и распределения прибыли

Бухгалтерия для типографии: приход-расход, затраты, канбан-доска заказов и
распределение прибыли между партнёрами. Валюта — киргизский сом (KGS).

## Что умеет

- **Панель управления** — продажи, себестоимость, валовая и чистая прибыль за
  месяц, остаток в кассе, долг клиентов, график за 12 месяцев
- **Приход-расход** — все движения денег, фильтры по категории и датам, итоги
- **Заказы (канбан)** — `Новый → Дизайн → Печать → Постпечать → Готов → Выдан`,
  перетаскивание карточек между этапами. У каждой позиции своя себестоимость и
  цена продажи — прибыль заказа видно сразу в форме
- **Задачи** — отдельная доска повседневных дел (`Сделать → В работе → Готово`)
  с приоритетом и сроком. Не связана с деньгами и заказами
- **Клиенты** — количество заказов, оплаченные суммы и задолженность
- **Категории** — виды приходов и расходов
- **Партнёры** — доли в процентах (в сумме ровно 100%), баланс долга по каждому
- **Распределение прибыли** — делит чистую прибыль месяца по долям, отслеживает выплаты
- **Отчёты** — произвольный период в разрезе категорий, экспорт в CSV
- **Пользователи** — три роли: администратор / бухгалтер / наблюдатель

## Запуск

### 1. MongoDB

**Atlas (бесплатное облако):** [cloud.mongodb.com](https://cloud.mongodb.com) →
создайте кластер M0 → добавьте пользователя в Database Access → разрешите
`0.0.0.0/0` в Network Access → Connect → Drivers → скопируйте строку подключения.

> **Внимание:** в строке от Atlas нет имени базы (`...mongodb.net/?appName=...`).
> Допишите его вручную после хоста, иначе всё запишется в служебную базу `test`:
>
> ```
> mongodb+srv://user:пароль@cluster0.xxx.mongodb.net/account?retryWrites=true&w=majority
> ```                                                     ↑ вот сюда

**Или локально:** `winget install MongoDB.Server`, строка: `mongodb://127.0.0.1:27017/account`

### 2. Сервер

```bash
cd server
npm install
cp .env.example .env      # подставьте свой MONGO_URI
npm run seed              # демо-данные (по желанию)
npm run dev               # http://localhost:5000
```

**Начать с чистого листа.** `npm run clean` удаляет демо-данные, но оставляет
справочник категорий и одного администратора — под ним же и входить. Партнёров,
клиентов, заказы и движения денег вводите свои; нумерация заказов начнётся заново
с `P-0001`. Если администратора в базе не окажется, скрипт создаст
`admin@poligraf.kg` / `admin123`.

### 3. Клиент

```bash
cd client
npm install
npm run dev               # http://localhost:5173
```

Vite проксирует `/api` на `localhost:5000` — с CORS возиться не нужно.

### 4. Тесты (по желанию)

```bash
cd server
npm test        # 44 сквозные проверки на MongoDB в памяти
```

Устанавливать MongoDB не требуется — тест сам поднимает временную базу.
Проверяются аутентификация, роли, расчёт себестоимости и прибыли заказа, перенос
карточек канбана, признание прибыли по дате выдачи и вся логика распределения —
включая то, что расход по заказу и выплата партнёру **не** списываются дважды.

## Деплой на Vercel

Проект настроен на деплой **одним проектом**: сайт и API живут на одном домене.
Клиент — статика, API — серверлес-функция под `/api`. CORS не нужен, менять
адреса в коде не приходится. Вся конфигурация — в [vercel.json](vercel.json),
обработчик функции — в [server/api/index.js](server/api/index.js).

```
myapp.vercel.app          → сайт (client/dist)
myapp.vercel.app/api/...  → Express (server, серверлес)
```

**Шаги:**

1. **Atlas → Network Access → разрешите `0.0.0.0/0`.** У серверлес-функций
   Vercel динамические IP, фиксированного адреса для белого списка нет. Без этого
   функция не достучится до базы.

2. **Залейте проект в Git** (GitHub/GitLab/Bitbucket). Файл `.env` в репозиторий
   не попадёт — он в [.gitignore](.gitignore).

3. **Import Project** на [vercel.com](https://vercel.com) → выберите репозиторий.
   Build- и Output-настройки трогать не нужно — их задаёт `vercel.json`.

4. **Settings → Environment Variables** — задайте те же, что в `server/.env`:

   | Переменная | Значение |
   |---|---|
   | `MONGO_URI` | строка Atlas **с именем базы** (`...mongodb.net/account?...`) |
   | `JWT_SECRET` | длинная случайная строка |
   | `JWT_EXPIRES_IN` | `7d` (необязательно) |

   `NODE_ENV=production` Vercel выставляет сам. `PORT` и `CLIENT_URL` не нужны.

5. **Deploy.** После сборки откройте домен — это готовый сайт.

6. **Первый вход.** База пустая, поэтому регистрация открыта: первый
   зарегистрированный becomes админом. Либо заполните демо-данными, запустив сид
   **локально** против той же базы Atlas:

   ```bash
   cd server
   MONGO_URI="<та же строка Atlas>" npm run seed
   ```

   (сид — это скрипт, на самой Vercel он не запускается).

**Альтернатива — через CLI**, без Git:

```bash
npm i -g vercel
vercel            # из корня проекта, дальше по подсказкам
vercel --prod     # продакшен-деплой
```

> Локально всё работает по-прежнему: `npm run dev` в `server/` и `client/`.
> Файл `server/api/index.js` нужен только Vercel — обычный запуск идёт через
> `server/src/index.js`.

## Как считается прибыль

Всё крутится вокруг одной формулы. Прибыль берётся **из заказов**, а не из
движения денег:

```
    Продажи      Σ цена продажи заказов, выданных за месяц
  − Себестоимость Σ себестоимость тех же заказов
  ─────────────────────────────────────────────────────────
  = Валовая прибыль
  + Прочий доход   приход БЕЗ привязки к заказу
  − Накладные      расход БЕЗ привязки к заказу (аренда, зарплата, свет)
  ─────────────────────────────────────────────────────────
  = Чистая прибыль  →  минус реинвестиция  →  делится по долям партнёров
```

Админ заводит заказ, указывая у каждой позиции **себестоимость** (`unitCost`) и
**цену продажи** (`unitPrice`). Сервер считает `costAmount` и `totalAmount` сам —
цифрам с фронтенда он не верит, потому что от них зависят выплаты партнёрам
([order.controller.js](server/src/controllers/order.controller.js)).

## Логика учёта (важно)

Эти решения приняты намеренно — прочитайте перед тем, как менять:

**Заказ попадает в прибыль по дате выдачи, а не оплаты.** Как только карточка
переходит в этап «Выдан», проставляется `deliveredAt` — по нему заказ и
относится к месяцу. Вернули карточку назад — дата снимается. Неоплаченный
выданный заказ прибыль всё равно принёс: деньги просто висят в долге клиента.

**Транзакция, привязанная к заказу, в прибыль не идёт.** Её сумма уже учтена в
себестоимости заказа. Такие транзакции двигают только кассу и показывают факт
против плана (`expenseAmount` против `costAmount`).

> ⚠️ **Поэтому материал, купленный для заказа, обязательно привязывайте к нему.**
> Расход без привязки считается накладным и вычтется из прибыли **второй раз** —
> поверх себестоимости. На странице «Прибыль» есть список «Движения вне заказов»
> — если материал попал туда, это видно сразу.

**Оплата заказа не вводится вручную.** `Order.paidAmount` и `expenseAmount`
вычисляются из привязанных транзакций
([orderTotals.js](server/src/services/orderTotals.js)) и пересчитываются при
создании, изменении и удалении транзакции. Долг по заказу не может разойтись с
кассой.

**Распределение прибыли — не расход.** При выплате партнёру транзакция расхода
**не создаётся** ([distribution.controller.js](server/src/controllers/distribution.controller.js)).
Распределение — это деление уже посчитанной прибыли, а не затрата. Если записать
выплату расходом, прибыль следующего месяца уменьшится дважды.

**Сумма долей — ровно 100%.** Если сумма `sharePercent` активных партнёров не
равна 100, распределение не создаётся. Остаток, потерянный при округлении,
добавляется к самой крупной доле — так сумма долей точно равна распределяемой
сумме ([profit.js](server/src/services/profit.js)).

**Тип категории обязан совпадать с типом операции.** К «приходу» нельзя привязать
категорию расхода — сервер отклонит.

**История замораживается.** В распределении сохраняются и доли партнёров, и весь
расчёт (`revenue`, `cogs`, `overhead`, `netProfit`) на момент создания. Через год
можно объяснить каждую цифру, даже если заказы задним числом отредактировали.

**Дашборд и «Прибыль» считают одной функцией.** `monthProfit()` вызывается с обеих
страниц — цифры на них разойтись не могут. Отдельно показывается **касса**: это
реальные деньги (весь приход минус весь расход), а не прибыль.

## API

Всё под `/api`, требуется заголовок `Authorization: Bearer <token>`
(кроме login и первичной регистрации).

```
POST   /auth/login              POST   /auth/register        GET  /auth/me
GET    /categories              POST/PATCH/DELETE /categories/:id
GET    /customers               POST/PATCH/DELETE /customers/:id
GET    /partners                POST/PATCH/DELETE /partners/:id      (админ)
GET    /orders                  GET  /orders/board
POST   /orders                  PATCH /orders/:id    PATCH /orders/:id/move
GET    /tasks                   GET  /tasks/board
POST   /tasks                   PATCH /tasks/:id     PATCH /tasks/:id/move
GET    /transactions            POST/PATCH/DELETE /transactions/:id
GET    /distributions           GET  /distributions/preview          (админ)
POST   /distributions           PATCH /distributions/:id/approve
POST   /distributions/:id/pay
GET    /reports/dashboard       GET  /reports/summary
```
